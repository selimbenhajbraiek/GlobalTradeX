Ce fichier décrit le modèle de données à représenter dans le diagramme de classes
de la Release 1. Il sert de consigne pour dessiner le diagramme (PlantUML, StarUML,
draw.io) ou pour le chapitre rapport PFE.

Release 1 = Sprint 1 « Gérer Compte » + Sprint 2 « Gérer Expéditions & Produits »

  • Sprint 1 : table users
  • Sprint 2 : tables shipments, products, shipment_products

Total à ce stade : 4 tables (+ énumérations métier)


--------------------------------------------------------------------------------
1. COMMENT LIRE CE DOCUMENT
--------------------------------------------------------------------------------

  • Chaque TABLE = une entité du diagramme (nom = nom de table MySQL).
  • PK = clé primaire | FK = clé étrangère | UK = unique
  • Les cardinalités sont indiquées en notation rapport :
        users (1) ──────< (0..*) shipments
        signifie : un utilisateur possède zéro ou plusieurs expéditions.

  • Les énumérations (UserRole, ShipmentStatus, etc.) peuvent être dessinées
    comme des types énumérés reliés par dépendance (..>) à la colonne concernée.

--------------------------------------------------------------------------------
4. LIENS ENTRE TABLES (Release 1)
--------------------------------------------------------------------------------

  (A) users ──< shipments  (propriétaire importateur)
      • shipments.owner_id → users.id
      • Cardinalité : 1 utilisateur → 0..* expéditions (owner)
      • ON DELETE CASCADE

  (B) users ──< shipments  (exportateur lié)
      • shipments.exporter_user_id → users.id
      • Cardinalité : 0..1 exportateur → 0..* expéditions
      • ON DELETE SET NULL

  (C) users ──< products
      • products.user_id → users.id
      • Cardinalité : 1 exportateur → 0..* produits catalogue
      • ON DELETE CASCADE

  (D) shipments ──< shipment_products ──> products
      • Table associative (many-to-many avec quantité)
      • 1 expédition → 0..* lignes ; 1 produit → 0..* lignes
      • Permet de lier le catalogue à une expédition sortante (exportateur)
